redis从3.0版本开始支持redis cluster。

数据分片

redis cluster没有使用一致性hash进行数据分片，而是采用了哈希槽的概念。redis cluster一共有16384（0~16383）个哈希槽，在部署集群之后，会将所有的哈希槽分成特定的区间。比如集群有三个节点A、B、C，则哈希槽将被分成3个区间：

  • 节点A包含从0到5500的散列槽。
  • 节点B包含从5501到11000的散列槽。
  • 节点C包含从11001到16383的散列槽。
    在存储数据时，对key进行CRC16运算然后对16384取模，根据余数落到哪个区间来决定到底将数据存放到哪个节点。

redis cluster主从模型

在redis集群中，如果一个节点失去响应，则整个集群将停止运行。为了在节点发生故障或无法与大多数节点通信时集群能正常运行，redis cluster使用了主从模型，当一个主节点发生故障时，集群将自动进行故障转移。

notice: 集群中的从节点不能作为其他节点的主节点，即主从复制无法嵌套。

redis cluster TCP端口

redis cluster的节点除了暴露给客户端的端口之外，还需要另一个端口来保持与其他节点的通信，此端口用来故障检测，配置更新，故障转移授权等，端口号等于前一个端口 + 10000。这个值是固定的，所以：

  1. 必须保证暴露给客户端的端口可访问；
  2. 必须保证客户端端口+10000可以访问。

redis cluster 部署参数配置

  • cluster-enabled\<yes/no>：只有以cluster方式运行的redis实例能加入到redis cluster中。
  • cluster-config-file\<filename>：节点集群配置信息文件路径。注意不能通过用户手动修改此文件信息，此文件由集群节点自己生成和维护，用来保存集群相关信息，当集群发生变化，节点会更新文件内容并保存到本地。
  • cluster-node-timeout\<milliseconds>：集群节点不可用超时时间。如果主节点超过指定时长无法访问，则集群会对其进行故障转移。
  • cluster-slave-validity-factor \<factor>：节点超时系数，集群将根据此来尝试对故障节点进行故障转移。例如，如果节点超时设置为5秒，而有效性因子设置为10，则从服务器与主服务器断开连接超过50秒将不会尝试对其主服务器进行故障转移。具体的计算方法为：(node-timeout * replica-validity-factor) + repl-ping-replica-period。
  • cluster-migration-barrier\<count>: 一个主节点将保持的最小的从节点数。当集群有一个主节点有超过此参数的从节点，而另一个主节点有少于此参数的从节点，则会尝试将多的从节点进行转移，以保持所有主节点都有至少此参数的从节点。
  • cluster-require-full-coverage \<yes/no>：当有主节点故障导致有的hash槽无法存储数据时，如果此选项被设置为 yes，则集群不会接受任何的读写操作。如果被设置为 no， 集群将依旧提供查询操作，即使部分的hash槽无法查询。

注意

redis集群只支持在一个节点中数据的多键操作，这意味着，如果你需要操作的数据分布在多个不同的节点上，那么你的多键操作将会失败。
为了解决这个问题，你可以使用一种 hash tag 的方法，来将需要多键操作的数据放在同一个节点上：redis cluster 在存放数据时，会先对key进行CRC16运算，并对16384取模，但是你可以使用 {} 符号强制只使用 key 的子串进行相关的运算。例如：key为one{FOO}和anoter{FOO}将会保存在同一个节点上，因为她只会对key的子串 ‘FOO’ 进行相关的运算。

在redis cluster 中，默认只有一个数据库，并且无法在节点中分库。
